---
title: 'GET /guests'
description: 'Get a filtered list of guests with advanced query options.'
---

<Note>
	This endpoint returns data in a **humanized format**, meaning all
	information is ready to be displayed in the language you requested. Use the
	optional `X-LOCALE` header for localization.
</Note>

#### Authentication

This endpoint requires authentication. See [Authentication](/api-reference/authentication) for detailed information about required headers and how to obtain your API keys.

#### Query Params

<ParamField path="startDate" type="Date" required>
	Filter guests by start date (from this date)
</ParamField>

<ParamField path="endDate" type="Date" required>
	Filter guests by end date (to this date)
</ParamField>

<ParamField path="role" type="'patient' | 'companion' | 'guest'">
	Filter guests by their role
</ParamField>

<ParamField
	path="status"
	type="'registered' | 'waitingList' | 'review' | 'confirmed' | 'arrived' | 'paid' | 'cancelled' | 'voided' | 'registeredOnline' | 'offer' | 'departed'"
>
	Filter guests by their booking status: - `registered` - Guest is registered
	- `waitingList` - Guest is on waiting list - `review` - Booking is under
	review - `confirmed` - Booking is confirmed - `arrived` - Guest has arrived
	- `paid` - Payment completed - `cancelled` - Booking cancelled - `voided` -
	Booking voided - `registeredOnline` - Registered through online system -
	`offer` - Like a draft booking, which can turn into a booking if the
	customer accepts the offer - `departed` - Guest has departed
</ParamField>

<ParamField path="start" type="string" default="1">
	The starting page number for pagination. Used to control which page of
	results to return.
</ParamField>

<ParamField path="length" type="string" default="50">
	The number of results to return per page.
</ParamField>

<ParamField path="project" type="array">
  Specify which fields to include in the response. You can project customer fields, booking fields, and custom meta fields:

**Customer Fields:**

-   `firstName`, `lastName`, `email`, `phone`, `secondaryPhone`
-   `dob`, `age`, `pin`, `gender`
-   `address.notes`, `address.city`, `address.zip`, `address.country`, `address.street`
-   `refferalType`, `religion`, `internalId`

**Booking Fields:**

-   `centerId`, `sessionId`, `sessionIds`, `programId`, `packageId`, `userId`
-   `status`, `role`, `isCompanionFor`, `practitionerIds`, `bookedAt`, `createdBy`
-   `startDate`, `endDate`, `bookingId`,
-   `medicalLetter.ultrasoundNotes`, `medicalLetter.ultrasoundStr`,
-   `medicalLetter.psychiatricExam`, `medicalLetter.diagnoses`, `medicalLetter.customRec`, `medicalLetter.ekg`, `medicalLetter.recommendationsIds`
-   `medicalData.notes`
-   `formData.medicalConditions`, `formData.observations`, `formData.mobility`, `formData.accomodationDetails`
-   `acc.roomId`, `acc.bedType`, `acc.type`, `acc.roomStandardId`, `acc.roomSize`, `acc.floor`, `acc.shareRoomWith`
-   `financial.subtotal`, `financial.discountIds`, `financial.total`
-   `financial.deposit.amount`, `financial.deposit.notes`, `financial.deposit.paidAt`, `financial.deposit.paymentType`
-   `financial.fullPayment.amount`, `financial.fullPayment.notes`, `financial.fullPayment.paidAt`, `financial.fullPayment.paymentType`

**Custom Meta Fields:**

-   `meta.<your-custom-meta-key>` - Replace with your actual custom meta field keys

### Finding Your Meta Keys

To find the available meta keys for your center:

1. Navigate to **General Settings** â†’ **Metas**
2. Search for the desired meta field
3. Copy the key from the **Key** column
4. Use it in the format `meta.<key-name>`

Example:

-   `project=firstName,lastName,meta.food-allergies`

<Note>
	If it's a meta you created, include the prefix `meta.` before the key name.
	If it's a customer or booking field, just use the field name directly.
</Note>

</ParamField>

#### Example Request

```bash
GET /guests?startDate=2025-01-01&endDate=2025-01-31&role=patient&status=confirmed&start=1&length=25&project=firstName,lastName,email,startDate,endDate,meta.food-allergies
```

#### Response

The response returns data in a **humanized format**, meaning all information is localized and ready to be displayed to end users in the requested language.

```json
{
	"data": [
		{
			"firstName": "John",
			"lastName": "Doe",
			"email": "john.doe@example.com",
			"startDate": "06/15/2024",
			"endDate": "06/22/2024",
			"status": "Confirmed",
			"role": "Patient",
			"ids": {
				"role": "patient",
				"status": "confirmed",
				"sessionId": "sess_123456",
				"userId": "user_789012",
				"bookingId": "booking_345678"
				// Other technical identifiers
			}
			// Additional fields based on project parameter
		}
	],
	"total": 125
}
```

<Note>
	**Date Formatting by Locale:** - **Romanian (ro):** DD.MM.YYYY format (e.g.,
	"15.06.2024") - **English and others:** MM/DD/YYYY format (e.g.,
	"06/15/2024")
</Note>

<Note>
	**Humanized vs Technical Data:** - **Main fields** (like `status`, `role`,
	dates) are returned in human-readable, localized format - **`ids` field**
	contains the technical/raw values needed for API operations and integrations
</Note>

<Note>
	**Pagination:** Use the `start` and `length` parameters to paginate through
	results. The response includes pagination metadata to help with navigation.
</Note>
